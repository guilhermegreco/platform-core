Why This Exists

The Problem We're Solving

In large enterprises, infrastructure expertise is distributed across specialized teams. The platform team can't own every abstraction — and shouldn't be the bottleneck for every capability.

Architecture

Three-Layer Separation of Concerns

Developer Layer

Writes values.yaml with a Score-like DSL. Declares capabilities needed (database, cache, events). Never touches infrastructure, security groups, IAM, or Kubernetes internals.

Platform Layer

Owns the platform-app Helm chart. Renders all Kubernetes resources from developer values. Controls Pod Identity, ABAC policies, readiness probes, and workload lifecycle.

Capability Layer

Independent teams (DB, Cache, Pub/Sub) own their ResourceGraphDefinitions. They define how AWS resources are provisioned, secured, and connected. Upgradeable without touching applications.

Developer Experience

One File. Everything Provisioned.

The developer writes a single values.yaml. The platform handles the rest.

# values.yaml - this is ALL the developer writes name: order-service team: commerce namespace: team-commerce image: 279051970617.dkr.ecr.us-east-1.amazonaws.com/order-service:v2.1.0 port: 8080 replicas: 2 database: enabled: true size: small cache: enabled: true size: small events: enabled: false

What Gets Created

ServiceAccount, IAM Role + Policy, Pod Identity Association, Deployment with readiness probe, Service, Database CR, Cache CR, connection Secrets - all from this one file.

Shared Resources

Need access to another team's database? Just declare class: existing with the resource ID. No tickets, no cross-team coordination. The policy scopes access automatically.

Each capability team owns their infrastructure abstraction end-to-end.

kro ResourceGraphDefinition

Each capability is a kro RGD that orchestrates multiple AWS resources into a single abstraction with proper dependency ordering and readiness checks.

SecurityGroup per Instance

Each capability instance creates its own SecurityGroup via ACK EC2 controller. No shared SGs, full lifecycle management, cleaned up on delete.

Independent Upgrades

Capability teams publish new chart versions. ArgoCD syncs the RGD. kro reconciles all existing instances. Zero developer action required.

What E2E Validates

Pod Identity credentials work. IAM policies allow Secrets Manager access. Real RDS connection established. Cache connected. Capability add/remove doesn't cause downtime. /ready returns 200.

Upgrade Safety

Deploy previous chart version first, then upgrade to current. Proves rolling updates don't break existing workloads. Old pod keeps serving until new one is Ready.

Quality

Testing Pyramid

Three tiers of validation — fast feedback at the bottom, full confidence at the top.

Tier What It Tests Where It Runs Time Catches
Tier 1 Chart renders valid YAML. RGD schema accepted by kro. CRD created. k3d + kro + CRDs only (no AWS) ~1 min Syntax errors, broken CEL expressions, invalid schema types
Tier 2 AWS resources actually provision. SecurityGroup, RDS, ElastiCache created and reach ACTIVE. k3d + kro + ACK controllers + real AWS APIs ~15 min IAM permission errors, wrong VPC/subnet, networking issues, ACK bugs
Tier 3 (E2E) Full workload lifecycle. Pod gets credentials, connects to DB, survives upgrades. Real EKS cluster (Pod Identity, ArgoCD) ~20 min Pod Identity race, ABAC policy denials, Secret name mismatches, upgrade regressions

Tier 1: Fast Feedback (every PR)

helm lint + helm template (all capability combos) + deploy RGD to k3d with kro. Validates schema, CEL expressions, template logic. No AWS credentials needed.

Tier 2: Real Infrastructure (on merge)

Capability Pipeline spins up k3d with real ACK controllers. ACK talks to AWS APIs from inside the CI runner. Creates real RDS/ElastiCache. Chainsaw asserts ACTIVE state. Tests upgrade path. Cleans up.

Tier 3: E2E Workload (on merge)

Platform App Pipeline deploys to real EKS cluster. Proves Pod Identity, IAM policy, Secrets Manager access, DB connectivity. Only this tier needs EKS (for Pod Identity webhook + managed ArgoCD). /ready is the assertion.

The Readiness Probe = The Test

The sample-app's /ready validates the entire chain: projected token → credentials endpoint → IAM role → Secrets Manager → DB password → RDS connection. If pod is 1/1 Ready, everything works.

CI/CD Deep Dive

Capability Pipeline Ownership & Dependencies

The platform team owns and maintains the shared CI/CD pipeline. Capability teams consume it via a thin caller workflow — they control what to test, not how to test it.

Platform Team Owns the Capability Pipeline

The Capability Pipeline and its internal CI Action live in platform-core. Capability teams invoke it but cannot modify the pipeline logic, job ordering, or tool versions. One pipeline governs all capabilities.

25-Line Caller = Team's Only Pipeline Code

Each capability repo has ONE workflow file: a thin workflow_call that passes inputs (chart path, ACK controllers, timeouts) to the Capability Pipeline. All complexity is abstracted away.

Governed Upgrade Path

When the platform team updates the Capability Pipeline (e.g., adds a new test phase, bumps tool versions), ALL capability teams get the change immediately on their next run — no per-repo PRs needed.

End-to-End Automation

The Capability Pipeline handles everything: test → version → publish to ECR → bump staging Chart.yaml → ArgoCD syncs to cluster. Capability teams just push code and the platform handles the rest.

CI/CD Deep Dive

Capability Pipeline CI Action Internals

The Capability Pipeline uses an internal action (setup-platform-ci) to bootstrap the test environment. Two modes — fake for fast PR feedback, real for integration confidence.

Why k3d (not Kind/minikube)?

k3d wraps k3s in Docker. Starts in ~10s, uses minimal RAM, supports multiple agents. The --no-lb and disabled traefik/metrics keep the CI runner lean — only what's needed for testing.

Why Pinned Versions?

Every tool version is set in the Platform CI Action, not in the caller. This prevents "works on my branch" issues. A version bump is a single PR to platform-core affecting all capability teams simultaneously.

Credential Flow (Real Mode)

GitHub OIDC → aws-actions/configure-aws-credentials → env vars. The Platform CI Action passes these into ACK controller pods via deployment.extraEnvVars. No long-lived secrets stored anywhere.

ACK Controller Selection

Each capability declares which controllers it needs (e.g., rds,ec2). Only those are installed — no wasted time. The database team doesn't install the ElastiCache controller and vice versa.

Delivery

Chart Promotion Flow

Everything is a Helm chart. Every output is a versioned OCI artifact in ECR. Promotion is a version bump in Git.

Capability Lifecycle: Tested → Published → Deployed to Staging

The capability pipeline lifecycle ends with the RGD deployed to the staging cluster. Here's the complete chain:

  1. Capability team pushes code to their repo (e.g., platform-capability-database)
  2. Pipeline tests the RGD with real AWS resources (provisions actual RDS, verifies ACTIVE, cleans up)
  3. On success: chart is packaged and published to ECR as a versioned OCI artifact (e.g., team-database:1.2.0)
  4. Pipeline auto-creates a Pull Request bumping the version in argocd/staging/Chart.yaml
  5. PR is auto-merged → ArgoCD detects the git change → syncs the new chart version → RGD is live in staging

The staging cluster always has the latest tested version of every capability RGD. This is the cluster where the platform-app E2E pipeline (Tier 3) validates that workloads can consume these capabilities end-to-end.

Automated: Staging

When a capability pipeline succeeds, it auto-creates a PR bumping the version in argocd/staging/Chart.yaml. The PR is auto-merged. ArgoCD syncs immediately. Zero human intervention.

Controlled: Production

Production promotion is a deliberate action. Platform team copies the proven staging versions to argocd/production/Chart.yaml. This is the release gate — one PR, reviewed and approved.

Everything is a Chart

Capabilities are charts. The platform-app is a chart. Even the staging environment is a chart (umbrella). The single unit of delivery across the entire platform is a versioned Helm chart in ECR.

Rollback = Version Revert

To rollback a capability: revert the version in Chart.yaml. ArgoCD syncs the old RGD. kro reconciles. No helm rollback, no manual intervention. Git is the source of truth.

Environment Strategy

Capability Chart.yaml Per Environment

Each environment has its own umbrella Chart.yaml pinning capability versions. One ArgoCD Application per environment deploys all RGDs.

One Chart.yaml Per Environment

Each environment pins its own capability versions. Staging is always ahead. Production environments can be at different versions during a gradual rollout.

Promotion = Version Bump PR

Promoting a capability from staging to dev is a one-line change in production/dev/Chart.yaml. Reviewed, approved, merged. ArgoCD syncs the new RGD version.

One Application Per Environment

ArgoCD has one Application per environment pointing to its Chart.yaml path. The ApplicationSet generates them from a list of environments. Each syncs independently.

Rollback = Revert Version

Bad capability in UAT? Revert the version in production/uat/Chart.yaml. ArgoCD syncs the previous RGD. kro reconciles. No manual cleanup needed.

Security

Least Privilege by Design

Dual-tag ABAC ensures workloads can only access their own resources.

Pod Identity (EKS)

Each workload gets its own IAM Role + PodIdentityAssociation. Credentials injected via projected serviceAccountToken volume - no webhook dependency, works on first deploy.

Dual-Tag ABAC

IAM policy conditions check both platform.io/workload AND platform.io/namespace tags. Cross-namespace access is impossible. Cross-workload access only with explicit class: existing declaration.

class: default (own resources)

Policy scoped to: aws:ResourceTag/platform.io/workload = "my-app" + aws:ResourceTag/platform.io/namespace = "team-commerce". Only your own resources.

class: existing (shared resources)

Policy scoped to: aws:ResourceTag/platform.io/workload = "shared-db" + aws:ResourceTag/platform.io/namespace = "team-commerce". Access only the declared shared resource, within your namespace.

Scenario Tag Condition Result -------- ------------- ------ app-a owns its DB workload=app-a AND namespace=team-commerce ALLOW app-b shares app-a's DB workload=app-a AND namespace=team-commerce ALLOW app-x (other namespace) workload=app-a AND namespace=team-payments DENY tries app-a's DB (resource tagged namespace=team-commerce) (mismatch)
